Codebase Folder Reorganization
What & Why
The project has grown to 135 flat server files (368 cross-imports) and 124 flat page files with no subdirectory grouping. This makes navigation slow, increases agent scanning overhead, and creates merge-conflict risk as the platform scales. Several route files also double as service layers — exporting core business logic consumed by other modules — which masks the real architecture.

Reorganizing into domain-based subfolders with proper separation of concerns improves developer productivity, organizational clarity, and reduces AI processing cost on every future task.

Firm Rule Going Forward: All new files must be placed in the appropriate domain subfolder — no more flat dumping into server/ or client/src/pages/.

Done looks like
Business logic extracted out of route files into proper service files (aeo-grader, custom-auth, engine-registry, admin-revenue)
server/routes.ts (1,642-line god file) renamed to server/router.ts and slimmed to a thin orchestrator importing ~10 domain registration functions instead of 89 individual files
Server files grouped into domain subfolders with @server/ path alias so imports survive future moves
Client pages grouped into domain subfolders
App.tsx (621 lines) refactored to use a route config array pattern instead of 100+ individual Route/lazy entries
Barrel index.ts files in each new subfolder re-exporting the public API
replit_integrations/ upward relative imports converted to use the new path alias
TypeScript compilation (tsc --noEmit), Vite production build (npm run build), and runtime smoke test all pass
STRUCTURE.md at project root documenting the folder convention
A lightweight structure validation script that flags new files landing directly in server/ or client/src/pages/ root
Out of scope
Renaming individual files (only moving them and extracting shared functions)
Refactoring business logic itself (only relocating it to proper service files)
Changing the shared/ or client/src/lib/ structure (already clean)
Reorganizing attached_assets/ (cosmetic, no code impact)
